# Верифікація функціональності MEDEVAK

**Дата:** 2025-03-17  
**Мета:** Перевірка відповідності системи двом основним інтерфейсам — бойовому (польовому) та командному (штабному).

---

## 1. Два інтерфейси — концепція

| Інтерфейс | Призначення | Користувач |
|-----------|-------------|------------|
| **Бойовий дашборд** | Швидке внесення даних на місці поранення | Медик у полі |
| **Командний дашборд** | Управління, архів, ресурси, координація | Штаб, координатор |

---

## 2. Бойовий дашборд / польовий екран

**Вимоги:** Максимальна швидкість, простота, великі кнопки, сенсорний екран, робота в рукавицях, поганому світлі, поспіху.

### 2.1 Функціональні вимоги vs поточний стан

| Вимога | Стан | Деталі |
|--------|------|--------|
| Швидко створити нового пацієнта | ⚠️ Частково | Є `POST /cases`, форма на dashboard. **Немає** окремого польового екрану з великими кнопками |
| Відмітити поранення на схемі тіла | ❌ Відсутньо | Backend: `injuries`, `body_markers` (API є). **Немає** UI зі схемою тіла (body map) |
| Базові дані: ім'я, позивний, підрозділ | ❌ Відсутньо | Case має лише `mechanism_of_injury`, `notes`, `triage_code`. **Немає** полів: `patient_name`, `call_sign`, `unit` |
| Час поранення | ❌ Відсутньо | Case має `created_at` (час створення запису). **Немає** `time_of_injury` |
| Час накладення турнікету | ❌ Відсутньо | API `tourniquets` — stub (повертає []). **Немає** моделі Tourniquet, часу накладення |
| Відмітити введені препарати | ✅ Є | `medications`, `POST /cases/{id}/medications` |
| Великі зрозумілі кнопки | ❌ Відсутньо | Поточний UI — стандартний desktop, дрібні елементи |
| Кнопка «плюс» для швидкого додавання | ⚠️ Частково | Є кнопка «Створити кейс» на сторінці Кейси. **Немає** універсальної кнопки «+» для подій/пацієнтів |
| Червона кнопка аудіозапису | ❌ Відсутньо | Не реалізовано |
| Голосове введення (Whisper API) | ❌ Відсутньо | PROJECT_STATE_AUDIT: «Voice / transcription / Whisper — Not found» |

### 2.2 Висновок по бойовому дашборду

**Логіка сформульована правильно**, але **реалізація неповна**:

- Backend підтримує частину даних (cases, injuries, medications, observations).
- **Відсутні:** окремий польовий UI, body map, поля пацієнта (ім'я, позивний, підрозділ), час поранення, турнікети, голосовий ввід.
- **Android app** (якби був) — природний носій бойового інтерфейсу (тач, офлайн, камера, мікрофон).

---

## 3. Командний дашборд / командний центр

**Вимоги:** Робота з масивом даних, архів, запаси, кров, координація.

### 3.1 Функціональні вимоги vs поточний стан

| Вимога | Стан | Деталі |
|--------|------|--------|
| Архів пацієнтів / кейсів | ✅ Є | `/cases` — список, `/cases/[id]` — деталі |
| Актуальний список пацієнтів | ✅ Є | `casesApi.list()` |
| Облік медичних запасів | ⚠️ Частково | Blood page фільтрує медикаменти за BLOOD_CODES. **Немає** окремого модуля inventory/stock |
| Облік складу медикаментів | ⚠️ Частково | `/medications` — введені препарати по кейсах. **Немає** залишків на складі |
| Контроль наявності крові | ✅ Є | `/blood` — фільтр продуктів крові |
| Перегляд, редагування, супровід записів | ✅ Є | Cases detail, PATCH case, medications, observations, procedures |
| Загальна координація | ⚠️ Частково | Sync, Audit, Handoffs. **Немає** єдиного «командного» екрану з агрегованим виглядом |

### 3.2 Висновок по командному дашборду

**Логіка відповідає вимогам**, реалізація **частково повна**:

- Є архів кейсів, handoffs, медикаменти, кров, аудит, синхронізація.
- **Відсутні:** окремий inventory (залишки на складі), єдиний командний екран (dashboard of dashboards).

---

## 4. Що втрачено або не відображено в архітектурі

| Елемент | Опис |
|---------|------|
| **Розділення інтерфейсів** | В документації не зафіксовано чітке розділення «польовий» vs «командний». Є один dashboard. |
| **Поля пацієнта** | `patient_name`, `call_sign`, `unit` — не в моделі Case |
| **Час поранення** | `time_of_injury` — не в Case |
| **Турнікети** | Модель Tourniquet, `time_applied` — не реалізовано |
| **Body map** | UI схеми тіла — відсутній |
| **Голосовий ввід** | Whisper API, аудіозапис — відсутні |
| **Inventory / stock** | Модель залишків медикаментів і крові — відсутня |
| **Польовий UX** | Великі кнопки, touch-friendly, high-contrast — не застосовано |

## 5. Рекомендації для цілісності системи

### 5.1 Оновити документацію

- Додати в `docs/architecture.md` розділ про **два інтерфейси** (польовий / командний).
- Зафіксувати в `PROJECT_STATUS.md` відсутні компоненти як backlog.

### 5.2 Розширити модель Case (міграція)

```
patient_name, call_sign, unit, time_of_injury
```

### 5.3 Додати модель Tourniquet

```
case_id, body_part_code?, time_applied, notes
```

### 5.4 Розділити dashboard (опційно)

- `/field` або `/intake` — польовий екран (великі кнопки, мінімум полів).
- Поточні `/cases`, `/handoffs`, `/blood` тощо — командний центр.

### 5.5 Backlog

- Body map UI
- Whisper / голосовий ввід
- Inventory (склад медикаментів і крові)
- Командний overview-екран

## 6. Підсумок

| Критерій | Оцінка |
|----------|--------|
| **Логіка сформульована правильно** | ✅ Так — два інтерфейси чітко описані |
| **Щось втрачено** | ⚠️ Так — поля пацієнта, час поранення, турнікети, body map, голос, inventory |
| **Структура цілісна** | ⚠️ Частково — командний дашборд близький до цілісності; бойовий потребує окремого UI та додаткових полів |

**Висновок:** Функціональна логіка двох інтерфейсів сформульована коректно. Реалізація покриває переважно командний центр; бойовий дашборд потребує окремого польового екрану, розширення моделей даних та голосового введення.
